# Proposal D3.5-Option3 — Process-Reward Verifier for cmaj Aggregation

**Author:** Phase-2 orchestrator | **Date:** 2026-05-03 | **Status:** Phase-3 candidate (not Phase-2-budget-eligible)

This is the third (and only remaining-open) architecture from the D3.5
verifier-based-aggregation family. Options 1 (TF-IDF + LR) and 2 (Qwen-0.5B
mean-pool encoder + MLP head) were tested in Phase-2 and both LOST. See
`phase2/spikes/verifier-aggregation/RESULT.md` for those negative results.

This option-3 variant is **architecturally different** — it scores branches
based on per-step trajectory features (entropy, commit-LoRA logit shifts,
mechanism source), not per-answer text/embedding features. It is gated on
upstream Workstream-C trace expansion (we have 1 real-mode STATUS-schema
trace; need 5-10 minimum to make a spike justified).

## Motivation

Both Phase-2 supervised verifiers failed at ~70-80% on the same dataset where
oracle ceiling is 88-90%. Diagnosis: per-branch surface features (text or
mean-pooled encoder embeddings) don't carry enough signal to discriminate
correct from incorrect arithmetic at our dataset scale.

Step-level features carry more information:
- **Per-position entropy**: high-entropy positions during diffusion correlate
  with model uncertainty about that token. Wrong-answer branches likely have
  higher entropy at the answer span.
- **Commit-LoRA logit shifts**: the commit-LoRA biases logits toward
  Phase-1-trained "good prefix" patterns. Branches where commit-LoRA
  disagrees strongly with the base model may be "rescued" or "corrupted" —
  worth distinguishing.
- **Mechanism source**: which sub-block was generated by LLaDA vs AR-extend
  vs cmaj-branch (per Workstream-C visualizer's manual mode). The handoff
  pattern itself may predict correctness.

This is process-reward in the PRM800K sense, but with sfumato-specific
trajectory features rather than human step-level annotations.

## Mathematical formulation

Each branch $y_i$ is now a sequence of $T$ sub-block records (typically $T=4$
for `gen_length=128, sub_block_length=32`). Each record has fields per the
Workstream-C STATUS-schema JSONL:

$$s_t^{(i)} = (\text{step\_idx}, \text{sub\_block}, \text{mechanism},
\text{tokens\_committed}, \text{entropy}, \text{commit\_lora\_active},
\text{logit\_shift\_norm})$$

The verifier scores branch $y_i$ as:

$$f_\phi(y_i) = \text{MLP}(\text{aggregate}(s_1^{(i)}, \ldots, s_T^{(i)}))$$

where `aggregate` could be:
- LSTM / transformer-encoder over the $T$ sub-block records (small, $T=4$ is tiny)
- mean-pool of per-step feature vectors
- summary statistics: mean entropy, commit-LoRA-active fraction, max logit_shift_norm

Inference rule unchanged: $\hat{a} = \text{extract}(y_{i^*})$ where
$i^* = \arg\max_i f_\phi(y_i)$.

## Quantitative success criteria (binding pre-reg if spike runs)

Same thresholds as options 1 and 2 (since they share the cmaj baseline):

| Outcome (held-out mean verifier accuracy) | Decision |
|---|---|
| $\geq 89\%$ | WIN-DECISIVE: D3.5 graduates to paper section, full experiment in Phase 3 |
| $\geq 87\%$ | WIN-STRONG: D3.5 graduates as paper section |
| $\geq 83\%$ | WIN-MINOR: D3.5 graduates as paper note |
| $\geq 78\%$ (parity) | INCONCLUSIVE: try architectural variants OR larger trace dataset |
| $< 78\%$ | LOSS: D3.5 dies in this option too; gap is genuinely uncloseable at this scale |

Per-step features have higher information content than mean-pool embeddings,
so a *parity* result with cmaj would be more interesting than option-2's
−8.5pp result — it would demonstrate that the gap CAN be partially exploited
by step-level signal, just not enough to win cleanly.

## Predictable failure modes

1. **Insufficient trace data**: with only 5-10 real-mode traces, training a
   process-reward MLP is severely under-determined ($\sim$50 sequences total).
   Mitigation: collect $\geq 50$ traces before spike, or use bootstrap
   averaging across many train/test splits.
2. **Trace data is too uniform**: all c2c condition runs without manual
   AR-extend / cmaj-branch interventions look the same — the mechanism-source
   feature degenerates. Mitigation: produce traces with varied manual
   interventions (mix of c2c, c3, cmajc, AR-handoff scenarios).
3. **Logit-shift-norm signal is null**: commit-LoRA logit shifts may not
   correlate with correctness if the shifts are small or if the LoRA was
   trained for trajectory-quality rather than answer-correctness.
4. **Step-level features still don't discriminate**: if process features fail
   too, the gap might be fundamentally unaddressable without ground-truth
   step-level supervision (à la PRM800K), and D3.5 dies in all 3 forms.

## Compute cost estimate

- **Trace collection** (Workstream-C real-mode runs, manual interactions on
  pod): 5-10 traces × 1 problem each × ~30s real-mode inference. ~$0.10
  total. Plus $\sim$2 hours of human time to drive the visualizer through
  diverse interaction patterns.
- **Verifier training**: tiny — sequence-encoder over $T=4$ records per
  branch, ~10K params. CPU-runnable, $\sim$1 minute.
- **Eval**: trivial.
- **Total spike compute: $\sim \$0.10$.**

## Dependencies

- ⛔ **Workstream-C trace expansion** — currently 1 trace
  (`phase2/inference_viz/traces/trace_c2c_real_mode_v3commit_problem42.jsonl`),
  but that's the runner.py `raw_*.jsonl` schema, not the STATUS-schema. Need to
  drive the visualizer's `server.py` through Gradio with manual interventions
  to produce STATUS-schema-compliant per-step traces.
- ⛔ **Logit-shift-norm instrumentation** — Workstream-C `server.py` currently
  always sets `logit_shift_norm=null`. Real-mode requires shadow base-adapter
  forward pass to compute the shift. Documented as a known limitation in
  `phase2/inference_viz/README.md`.

Both dependencies are deferrable but blocking until then.

## Why this proposal is not Phase-2-graduating

Three reasons:

1. **Substrate not ready**: STATUS-schema traces don't exist at scale yet.
2. **Phase-2 budget already scoped**: D1 (mode router) is the formally-promoted
   graduating proposal per `phase2/proposals/RANKING.md` (after D3.5 options
   1+2 lost). D1 itself depends on the same C-trace expansion, so option-3
   piggybacks on D1's substrate work.
3. **Honest publication bar**: another negative result on the same gap (option
   3 lost too) doesn't add scientific value over the existing 2 negative
   results in the Phase-2 paper section.

If D1 succeeds and produces useful real-mode trace data as a byproduct, this
proposal becomes Phase-3-spike-eligible.
